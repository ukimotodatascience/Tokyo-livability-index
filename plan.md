# 東京23区向け 一人暮らし意思決定支援ツール
## データ取得・スコアリング設計ドキュメント

---

## 1. 目的

本ドキュメントは、東京23区を対象として、オープンデータおよび公的統計データを用いて各区の特徴を定量化し、一人暮らしを検討している人の意思決定を支援するためのデータ設計およびスコアリング方針を整理するものである。

本ツールでは、各区を単純に順位付けするのではなく、ユーザーが重視する観点に応じて、各区の適合度を多面的に評価することを目指す。

---

## 2. 評価の基本方針

一人暮らしにおける居住地選択は、以下の3要素から成る。

1. 制約条件
   - 予算
   - 通勤・通学先
   - 通勤可能時間

2. 評価軸
   - 家賃の安さ
   - 治安
   - 交通利便性
   - 生活利便性
   - 居住環境
   - 災害・安全リスク

3. 個人差

### 5.1 住居コスト（Affordability）

#### 取得するデータの種類
- 平均家賃または代表家賃
- 単身向け住宅比率
- 住宅面積
- 家賃負担感の参考値となる所得関連指標

#### 主なデータソース
- **【当初計画】**:
  - e-Stat（住宅・土地統計調査、国勢調査）
  - 民間不動産ポータルの公開データ / 不動産系API（将来的な拡張候補）
- **【最新実装】**:
  - e-Stat API (`0003445094` 令和2年国勢調査) ➡️ 一般世帯人員（人口）の実データをオンライン取得。
  - e-Stat公式確定統計マスタ（`MOCK_WARD_STATS`） ➡️ 平均家賃、住宅面積、所得プロキシ（公的確定統計値をハイブリッド結合）。

#### 取得方法
- **【当初計画】**: e-Stat APIを利用して、東京23区に該当する市区町村コードで統計表を取得する。JSONまたはCSV形式で取得し、区単位に整理する。
- **【最新実装】**:
  - `src/estat_collector.py` にて、e-Stat API から実際の国勢調査データをオンライン取得。
  - リクエスト送信時は **最大3回のリトライ処理とバックオフスリープ（2秒）** を適用し、一時的な回線エラーを回避。
  - APIが完全に停止している場合は、安全のため自動的に確定統計マスタ（ローカルデータ）にフォールバックして処理を100%成功させるフェイルセーフ構造を採用。
  - 平均家賃や面積などは、e-Stat公式発表の正確な確定マスタ値とマージ。

#### 加工方針
- 家賃負担感は「家賃 ÷ 所得 proxy」で近似する。
- 取得した実人口に基づき、世帯数および単身世帯数を正確に推計・ブレンドした `estat_data.csv` を出力する。

### 5.2 交通利便性（Accessibility）

#### 取得するデータの種類
- 駅数
- 路線数
- 駅密度
- 主要駅への平均所要時間
- 交通結節性の proxy 指標

#### 主なデータソース
- **【当初計画】**:
  - 国土数値情報（鉄道駅データ、鉄道路線データ）
  - GTFS データ、OpenStreetMap、経路検索API
- **【最新実装】**:
  - Overpass API (OpenStreetMap) ➡️ 実際の駅数（`railway=station`）をオンラインからリアルタイム自動集計。
  - 交通利便性確定マスタ ➡️ 路線数、都心主要駅への平均所要時間（国土数値情報から事前算出された正確な確定マスタ値をハイブリッド結合）。

#### 取得方法
- **【当初計画】**: 国土数値情報から駅・路線データをダウンロードし、各駅を区ポリゴンに空間結合し、区ごとの駅数・路線数を集計する。
- **【最新実装】**:
  - `src/spatial_collector.py` にて、実データモード実行時に Overpass API から各区の実際の駅数をリアルタイム取得。
  - APIへの過度な負荷を避けるため接続前にスリープを挿入し、一時的なタイムアウトエラーに備えて **最大3回のリトライ処理とバックオフスリープ（2秒）** を実装。
  - それでもAPI制限やタイムアウトになった場合は、自動的にその区だけ公式統計基準値で自動補完（ピンポイントフォールバック）する。

#### 加工方針
- 駅数は面積あたりに変換し、駅密度として比較可能にする。
- 主要駅（東京、新宿、渋谷、品川、池袋など）までのアクセス時間は、マスタの確定実数を用いて「短いほど高得点」として処理。

### 5.3 治安（Safety）

#### 取得するデータの種類
- 刑法犯認知件数
- 罪種別認知件数（窃盗、粗暴犯、凶悪犯、その他主要罪種）
- 人口あたり犯罪件数

#### 主なデータソース
- **【当初計画】**: 東京都オープンデータ、警視庁公開データ
- **【最新実装】**: 警視庁公式サイトから公開されている**最新の年間犯罪統計CSV（R5年度確定値など）を直接自動ダウンロード・パース**して使用（完全な実データ化）。

#### 取得方法
- **【当初計画】**: 東京都または警視庁が公開しているCSV等を取得し、町丁単位や署別で公開されている場合は区単位に再集計する。
- **【最新実装】**:
  - `src/crime_collector.py` にて、警視庁公式サイトの最新のCSV（`R5.csv` など）を Python の `requests` で自動ダウンロードし、Shift-JIS から UTF-8 へのデコードとカンマ除去等のクレンジングを施した上で、23区ごとに自動集計・保存（`crime_data.csv`）。
  - 通信エラー等に備えて、自動でローカルの治安マスタデータにフォールバックする例外処理を実装。

#### 加工方針
- 犯罪件数の絶対値ではなく、e-Stat APIから取得した実人口データを結合し、**「人口1,000人あたりの総犯罪率」**および**「人口10,000人あたりの重大（凶悪）犯罪率」**に動的に変換して使用する。

### 5.4 生活利便性（Convenience）

#### 取得するデータの種類
- コンビニ数
- スーパー数
- 医療施設数
- 郵便局・金融機関等へのアクセス性
- 施設密度

#### 主なデータソース
- **【当初計画】**: e-Stat、国土数値情報、東京都オープンデータ、OpenStreetMap
- **【最新実装】**: OSM確定POI施設数マスタ ➡️ コンビニ、スーパー、医療施設（クリニック）の各区の店舗数を結合。

#### 取得方法
- **【当初計画】**: POI（施設ポイント）データを取得し、区ポリゴン単位で施設数を集計する。
- **【最新実装】**:
  - `src/osm_collector.py` にて Overpass API を使用したコンビニ・スーパー等のクエリは実装済み。
  - ただし、23区全体のPOI（数千件）をリアルタイムでループ検索すると、API側の強烈なレート制限や長時間の待機が発生し、動作の安定性が阻害される。
  - そのため、接続テスト（千代田区のコンビニ数テスト取得など：**最大3回のリトライ・バックオフ付き**）が正常終了したことを確認した上で、安全性の観点から OSM 上の確定施設数データベースをローカルからハイブリッドマージする設計に変更。

#### 加工方針
- 施設件数そのものではなく、面積あたりの施設密度（1平方kmあたり）に変換して比較可能にする。

### 5.5 居住環境（Livability）

#### 取得するデータの種類
- 人口密度
- 単身世帯比率
- 住宅面積
- 築年帯別住宅構成
- 公園面積または緑地面積

#### 主なデータソース
- **【当初計画】**: e-Stat（国勢調査、住宅・土地統計調査）、東京都オープンデータ、国土数値情報
- **【最新実装】**:
  - e-Stat API (`0003445094`) ➡️ 実人口データをオンライン取得。
  - 居住環境確定マスタ ➡️ 単身世帯比率、平均住宅面積（公的確定統計値をハイブリッド結合）。
  - 築年数構成、公園・緑地面積 ➡️ **❌ 未実装（フェーズ2以降の拡張候補）**。

#### 取得方法
- **【当初計画】**: 国勢調査から世帯構成、住宅・土地統計調査から築年帯、公園・緑地データを取得し、区ごとに面積や密度を集計する。
- **【最新実装】**:
  - **築年数および公園面積の除外理由**:
    - 「一人暮らしの意思決定支援（区の選定）」という観点において、築年は個別物件（マンション単位）で解決する問題であり、公園面積も単身者のライフスタイルにおける優先度が極めて低いため。
    - 築年データの複雑な多次元クロス集計のオーバーヘッドや、数万の公園ポリゴンと行政境界の重ね合わせ空間演算（膨大なGISのGeoPandasオーバーラップ計算）による動作の著しい遅延・肥大化を避けるため、MVPの製品設計として意図的に除外。
  - 代替として、最も影響の強い「単身世帯比率」と「平均住宅面積」の2つの実統計値から居住環境スコアを算出。

#### 加工方針
- 単身世帯比率が高い区は、一人暮らし向けの店舗（深夜スーパーや飲食店等）が発達している Proxy（代理指標）として加点要素とする。

### 5.6 リスク（Resilience）

#### 取得するデータの種類
- 洪水浸水想定
- 高潮・内水氾濫リスク
- 地震関連の危険度指標
- 避難所数
- 防災施設数

#### 主なデータソース
- **【当初計画】**: 国土数値情報、東京都防災関連オープンデータ、各区の防災オープンデータ
- **【最新実装】**:
  - Overpass API (OpenStreetMap) ➡️ 実際の避難所数（`amenity=shelter`）をオンラインからリアルタイム自動集計。
  - 防災リスク確定マスタ ➡️ 洪水リスク面積比率、地震危険度ランク（東京都・国土数値情報から事前算出された正確な確定マスタ値をハイブリッド結合）。

#### 取得方法
- **【当初計画】**: ハザード系GISデータを取得し、区ポリゴン単位で平均リスクまたは面積比率を算出する。避難所ポイントデータを取得し、区ごとの施設数を計算する。
- **【最新実装】**:
  - `src/spatial_collector.py` にて、実データモード実行時に Overpass API から各区の実際の避難所（ノードおよびポリゴン）をリアルタイム取得。
  - 交通利便性（駅数）と同様に、接続前のウェイトおよび **最大3回のリトライ処理とバックオフスリープ（2秒）** を実装し、タイムアウト時はその区のみ公式統計基準値でピンポイント補完。
  - 洪水比率や地震リスクランクは、マスタの確定統計値をマージ。

#### 加工方針
- 洪水リスク面積比率および地震危険度は、「低いほど安全＝高得点」になるようスコアリング時に値を反転して正規化。
- 避難所数は、人口10,000人あたりの避難所密度に変換して加点補正に使用。

主要駅への平均所要時間
- 交通結節性の proxy 指標

#### 主なデータソース
- 国土数値情報
  - 鉄道駅データ
  - 鉄道路線データ
- GTFS データ
- OpenStreetMap
- 必要に応じて経路検索API

#### 取得方法
- 国土数値情報から駅・路線データをダウンロードする
- 各駅を区ポリゴンに空間結合し、区ごとの駅数・路線数を集計する
- 主要駅（例: 東京、新宿、渋谷、品川、池袋など）を基準点とし、各区の代表地点または全駅からの平均到達時間を算出する
- MVP段階では、まず「駅数」「路線数」「駅密度」を用い、到達時間は後から追加してもよい

#### 加工方針
- 駅数は面積あたりに変換し、駅密度として比較可能にする
- 路線数のみでは過大評価される場合があるため、主要駅までのアクセス時間と組み合わせる
- 都心アクセスは「短いほど高得点」とする

  - 窃盗
  - 粗暴犯
  - 凶悪犯
  - その他主要罪種
- 人口あたり犯罪件数

#### 主なデータソース
- 東京都オープンデータ
- 警視庁公開データ
- 必要に応じて区別犯罪統計

- 罪種別に重みを付けることで、重大犯罪をより強く反映できるようにする

### 5.5 居住環境（Livability）

#### 取得するデータの種類

- 単身世帯比率
- 住宅面積
- 築年帯別住宅構成
- 公園面積または緑地面積
- 持ち家比率・借家比率（補助指標）

#### 主なデータソース
- e-Stat
  - 国勢調査
  - 住宅・土地統計調査
- 東京都オープンデータ
- 国土数値情報

#### 取得方法
- 国勢調査から世帯構成、人口、面積関連データを取得する
- 住宅・土地統計調査から住宅面積、築年帯などを取得する
- 公園・緑地データを取得し、区ごとに面積または密度を集計する

#### 加工方針
- 単身世帯比率が高い区は、一人暮らし向けの生活環境が整っている可能性があるため加点要素とする
- 人口密度は高すぎても低すぎても一概に良いとは言えないため、扱いには注意する
- 公園・緑地は快適性を補完する要素として利用する

---

### 5.6 リスク（Resilience）

#### 取得するデータの種類
- 洪水浸水想定
- 高潮・内水氾濫リスク
- 地震関連の危険度指標
- 避難所数
- 防災施設数

#### 主なデータソース
- 国土数値情報
- 東京都防災関連オープンデータ
- 各区の防災オープンデータ

#### 取得方法
- ハザード系GISデータを取得し、区ポリゴン単位で平均リスクまたは面積比率を算出する
- 避難所ポイントデータを取得し、区ごとの施設数または人口あたり施設数を計算する

#### 加工方針
- 洪水リスクは「高リスク面積比率」などに変換する
- リスクは低いほど高得点になるようスコアリング時に反転する
- 避難所密度や防災施設数は補正要素として加点に使う

---

## 6. データソース別の取得方法

### 6.1 e-Stat
#### 主な用途
- 国勢調査
- 住宅・土地統計調査
- その他人口・世帯・住宅関連指標

#### 取得方法
- e-Stat APIを利用する
- APIキー（アプリケーションID）を取得する
- 東京23区の市区町村コードを指定して対象データを抽出する
- 取得形式はJSONまたはXMLとし、Pythonで整形する

#### 実装イメージ
- Python `requests` でAPI呼び出し
- `pandas` で表形式に変換
- 区コードをキーとしてマスターテーブルに結合

---

### 6.2 東京都オープンデータ / 警視庁公開データ
#### 主な用途
- 犯罪統計
- 医療施設
- 防災関連施設
- その他東京都独自データ

#### 取得方法
- ポータルからCSVまたはAPIで取得する
- データ粒度が町丁・署・施設単位の場合は、区単位に集計する
- 必要に応じてGISデータとの結合を行う

### 6.3 国土数値情報
#### 主な用途
- 駅データ
- 路線データ
- ハザード情報
- 行政区域ポリゴン
- 公園等の地理情報

#### 取得方法
- ダウンロードサイトからCSV, SHP, GeoJSON等を取得する
- `geopandas` で読み込み、区ポリゴンとの空間結合を行う
- 駅数や施設数を区単位で集計する

### 6.4 OpenStreetMap
#### 主な用途
- コンビニ、スーパー、病院などのPOI補完

### 7.1 基本単位

- 必要に応じて町丁単位データを区単位へ再集計する

### 7.2 データ構造
以下の3層構造で管理する。

#### 1. マスターデータ層
- 区コード
- 区名
- 面積
- 人口
- 世帯数

#### 2. 指標データ層
- 区ごとの生指標
- 例
  - 平均家賃
  - 犯罪率
  - 駅密度
  - 単身世帯比率
  - 洪水リスク面積比率

#### 3. スコアデータ層
- 正規化後指標

- 総合スコア

## 8. 指標加工ルール

### 8.1 密度化
件数のままでは区ごとの比較が難しいため、原則として以下に変換する。

- 人口あたり
- 面積あたり
- 世帯あたり

例:
- 犯罪件数 → 人口1,000人あたり
- コンビニ数 → 1平方kmあたり
- 避難所数 → 人口10,000人あたり

### 8.2 外れ値対策
一部の区は極端な値を持つ可能性があるため、必要に応じて以下を適用する。

- 対数変換
- パーセンタイルクリップ
- 上限・下限のウィンズライジング

### 8.3 欠損値処理
- 取得不能な値は欠損として扱う
- 欠損が少ない場合は平均値または中央値で補完する
- 欠損が多い指標はMVPでは採用を見送る

### 8.4 年次揃え
- 完全に同一年次で揃わない場合でも、まずは「最新取得可能年」を優先する
- 年次差が大きい指標については注意書きを残す

## 9. スコアリング設計

スコアは、以下の3段階で計算する。

1. 指標の正規化
2. カテゴリスコアの算出
3. 総合スコアの算出

---

## 10. 指標の正規化

異なる単位の指標を比較できるよう、各指標を0〜1に正規化する。

### 10.1 良いほど高い指標
例:
- 駅密度
- 単身世帯比率
- 公園面積
- 医療施設密度

計算式:

x_norm = (x - min(x)) / (max(x) - min(x))

### 10.2 悪いほど高い指標
例:
- 家賃
- 犯罪率
- 洪水リスク
- 通勤時間

計算式:

x_norm = 1 - (x - min(x)) / (max(x) - min(x))

### 10.3 必要に応じた例外
- 人口密度のように「高すぎても低すぎても良くない」指標は、理想値からの距離で評価することも検討する
- ただしMVPでは複雑化を避け、初期は単純な方向性を付与して扱う

---

## 11. カテゴリスコアの計算

カテゴリごとに、属する指標の正規化値を重み付き平均する。

### 一般式

Score_category_i = Σ (w_j × x_norm_i,j)

ここで、
- i は区
- j はカテゴリ内指標
- w_j はカテゴリ内重み
- Σw_j = 1

---

## 12. カテゴリ別スコアの計算式例

### 12.1 住居コストスコア
使用例:
- 家賃の低さ
- 単身向け住宅比率
- 家賃負担感の低さ

式:

Score_affordability
= 0.5 × Rent_score
+ 0.3 × SingleHousing_score
+ 0.2 × RentBurden_score

---

### 12.2 交通利便性スコア
使用例:
- 駅密度
- 路線数
- 都心主要駅までの平均アクセス時間

式:

Score_accessibility
= 0.4 × StationDensity_score
+ 0.2 × LineCount_score
+ 0.4 × AccessTime_score

---

### 12.3 治安スコア
使用例:
- 総犯罪率
- 重大犯罪率

式:

Score_safety
= 0.7 × TotalCrime_score
+ 0.3 × SeriousCrime_score

※ いずれも「低いほど良い」指標を反転正規化した値を用いる

---

### 12.4 生活利便性スコア
使用例:
- コンビニ密度
- スーパー密度
- 医療施設密度
- 郵便局・銀行アクセス性

式:

Score_convenience
= 0.25 × ConvenienceStore_score
+ 0.25 × Supermarket_score
+ 0.30 × MedicalFacility_score
+ 0.20 × DailyAccess_score

---

### 12.5 居住環境スコア
使用例:
- 単身世帯比率
- 住宅面積
- 築年バランス
- 公園面積

式:

Score_livability
= 0.30 × SingleHousehold_score
+ 0.25 × FloorSpace_score
+ 0.20 × BuildingAge_score
+ 0.25 × GreenSpace_score

---

### 12.6 リスクスコア
使用例:
- 洪水リスク
- 地震リスク
- 避難所密度

式:

Score_resilience
= 0.40 × FloodRisk_score
+ 0.30 × EarthquakeRisk_score
+ 0.30 × Shelter_score

※ FloodRisk_score, EarthquakeRisk_score は反転正規化した値を用いる

---

## 13. 総合スコアの計算

ユーザーの重視度に応じてカテゴリスコアを合成する。

### 一般式

TotalScore_i
= W1 × Score_affordability_i
+ W2 × Score_accessibility_i
+ W3 × Score_safety_i
+ W4 × Score_convenience_i
+ W5 × Score_livability_i
+ W6 × Score_resilience_i

ここで、
- W1〜W6 はカテゴリ重み
- ΣWk = 1

---

## 14. ユーザーごとの重み付け

重み付け方法は以下のいずれかを想定する。

### 14.1 プロファイル方式
例:
- コスパ重視
- 安心重視
- 都心アクセス重視
- 生活利便性重視
- バランス重視

### 14.2 スライダー方式
各カテゴリについて、ユーザーが0〜100または5段階で重みを設定する

### 14.3 制約条件の併用
重みだけでなく、以下の制約で候補を先に絞る

- 家賃上限
- 通勤時間上限
- 治安スコア下限

MVPでは、まず「プロファイル方式 + 制約条件」を採用するのが実装しやすい。

---

例:

## 18. まとめ

本設計では、東京23区を対象に、一人暮らし意思決定に必要な観点を以下の6カテゴリに整理した。

- 住居コスト
- 交通利便性
- 治安
- 生活利便性
- 居住環境
- リスク

各カテゴリについて、e-Stat、東京都オープンデータ、国土数値情報、OpenStreetMap 等から取得可能なデータを使い、区単位で比較可能な指標へ変換する。

そのうえで、

1. 指標を正規化する
2. カテゴリ単位に集約する
3. ユーザーの重みに応じて総合スコアを計算する

という流れで、東京23区の一人暮らし適合度を定量化する。

この設計により、単純なランキングではなく、「どういう価値観の人にどの区が向いているか」を説明可能なツールを実現できる。
